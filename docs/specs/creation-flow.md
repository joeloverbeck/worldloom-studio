# Creation Flow Spec

This spec defines Worldloom Studio's first guided flow: `05_creation_protocol.md` phases 1-2 plus the non-generative Minimal Viable World checkpoint for phases 4-8 after seed evidence has been admitted. It is downstream of `docs/worldbuilding-system/05_creation_protocol.md`, `docs/worldbuilding-system/20_ai_assisted_workflow.md`, `docs/worldbuilding-system/21_templates_index.md`, `docs/worldbuilding-system/22_glossary.md`, `docs/worldbuilding-system/operating_card.md`, the foundational principles, ADRs 0001-0004, and `docs/specs/schema-v1.md`.

The first terminal state is a living world kernel plus seed records parked at `proposed`. A saved `world_kernel` alone proves Creation has begun; it does not complete Creation phases 1-2 and does not earn Admission queue work. Until seed decomposition parks proposed seeds, the workflow map keeps Creation active or owed for seed decomposition, keeps Admission unavailable with proposed-seed unlock guidance, and routes the next decision back to Creation. If the server ever cannot park seeds, it must return an explicit no-seed blocked state rather than treating Admission as done. After Admission admits seed facts, Creation also owns a Minimal Viable World checkpoint that reads accepted seed evidence, records steward dispositions in append-only checkpoint `pass_report` snapshots, and routes fact-shaped follow-up to Admission at `proposed`. This flow does not admit seeds, run the frontloaded seed audit, assign severity, produce gate results, or compute a pass/fail world verdict.

PRD #150 retrofits this flow into a W-8 decision-point surface for new-world start/resume, kernel authoring, Creation-bound Prompt-out, and seed decomposition. The durable parity source is `reports/app-parity-trial-01-creation-decision-points.md`; any temporary/local field-trial transcript that preceded that report is not stable implementation authority unless tracked.

PRD #165 extends the already-shipped phases 1-2 path after the Trial 03 field trial in `reports/app-parity-trial-03-replacement-guidance-prompt-context.md`. That report validates setup/open from PRD #158 and seed parking from PRD #150, while naming the remaining gap: after seed parking, the browser and prompt packet must be a replacement-grade Creation-to-Admission handoff rather than a list of markdown source paths or a kernel-only decomposition prompt. The temporary Trial 03 source was summarized for PRD #165 and must not be cited as stable implementation authority except through this tracked report or later durable issue comments.

PRD #222 is the Field Build 01 repair for the kernel-to-decomposition handoff. The field build proves that representative walkthrough evidence was too broad for the exact first-seed path: the browser must expose the consequence-mode control, truth-layer control, selected-section kernel guidance, selected-section Prompt-out grain, decomposition readiness, inline recovery, and workflow-map state grammar in the routed flow itself before Creation phases 1-2 can claim the handoff is usable with the methodology docs closed.

## Deferred Decisions

1. **Kernel section headings.** `record_section_headings` seeds the world kernel headings from `templates/world_kernel.md`: World premise; Core promise; Starting scale; Genre, tone, and consequence-mode commitments; Foundational facts; Foundational constraints; Initial mysteries and protected effects; Primary pressures and initial domains; Ordinary-life promise. `05_creation_protocol.md` constrains their meaning and wins on conflict.
2. **Seed decomposition report shape.** The append-only `seed_decomposition` report uses sections for Kernel source, Drafts consumed, Granularity decisions, Parked seeds, and Thin-start boundary, all constrained by `05` Phase 2.
3. **Draft-space data model.** Drafts live in a `drafts` table inside the world file. They are not records, owe no truth layer or canon status, are absent from record listings/search/link targets, and leave draft space only through explicit conversion or decomposition.
4. **Prompt-context assembly.** A prompt-out step assembles the selected record's title, type, truth layer, canon status, body, section content, role framing, vocabulary guardrail, label-assumptions instruction, and standing rulings. For `decomposition_pressure` after seed parking, the selected material is the seed-decomposition report and parked seed records, with the kernel only supporting context.
5. **Default prompt derivations.** Default prompts are versioned rows derived from `20` analyst roles. The original `20` text remains visible; steward edits create a new version; revert creates a new version equal to the original.
6. **Flow-instance persistence.** `flow_instances` stores flow key, current step, state, and kernel record id in the world file so reopening resumes the parked step.
7. **Operating-card derivation.** The UI displays a persistent operating-card panel derived from `operating_card.md`, with the upstream source named as provenance. Re-checking is manual in this slice: the derivation states its source and versioned text is updated with the app. Source paths are audit detail; the steward-facing card carries the method instruction needed at the current decision point.
8. **Skip records.** Declining an offered prompt writes a `skip_record` mechanically. Below the major-fact threshold, reason text is not required and the record states that no reason was collected.
9. **Consequence-mode representation.** The world kernel slot remains prose, but the authoritative consequence-mode declaration is the `consequence_mode` facet. The flow never defaults it; the steward must choose it explicitly when that step is completed.
10. **Minimal Viable World checkpoint.** The phases 4-8 checkpoint is a non-generative Creation decision point over admitted seed facts. The app clerks existing evidence for ordinary-life residue, adapted institution/custom, factional or mode-equivalent pressure, path-dependence residue, mystery boundary, aesthetic rule/residue, and pressure line. The steward records each dimension as covered, deferred, or protected mystery; deferrals create governed `skip_record` or `canon_debt` records. The checkpoint writes append-only `pass_report` snapshots, never edits admitted seed records, never changes canon standing, and never computes a pass/fail verdict.

## Step Map

| Step | Package source | Decision point | Dependency-bearing | Severity path | Prompt-out modes and pressure role |
|---|---|---|---|---|---|
| Kernel authoring | `05_creation_protocol.md` Phase 1 and `templates/world_kernel.md` | Define the first governing kernel or pressure seed, including explicit consequence mode. | yes | Not severity-scaled; consequence mode is steward-declared and never defaulted. | Proposal and pressure modes use `kernel_pressure`; proposal mode is available for empty non-premise selected sections, pressure requires steward-authored section material, and the World premise section keeps the server-owned essence exception/blocker by selected or requested target heading. |
| Seed decomposition | `05_creation_protocol.md` Phase 2 | Split broad steward material into seed facts that can be independently rejected, then park them at `proposed`. | yes | Not severity-scaled; Admission later owns severity. | Proposal and pressure modes use `decomposition_pressure`; pressure role is Prerequisite auditor over the seed-decomposition report plus parked seeds after decomposition material exists. |
| Minimal Viable World checkpoint | `05_creation_protocol.md` Phases 4-8 and `20_ai_assisted_workflow.md` | Judge whether admitted seed facts form a world-shaped pressure field by recording per-dimension dispositions. | yes | Not severity-scaled; it may create skip/debt records whose own reason thresholds follow shared skip policy. | Proposal and pressure modes use `minimal_viable_world_checkpoint`; proposal mode can suggest labeled residues/debt phrasing, and pressure mode challenges backdrop-shaped gaps only after steward material exists. |

Pressure prompts preserve `20`'s authored-material sequence: the steward's material is present first, and the model is asked for pressure, not first-draft canon. Proposal prompts preserve `20`'s proposal sequence instead: the steward fixes the judgment and context first, then asks for labeled candidate material. For Creation kernel authoring, that means an empty non-premise selected section may request proposal candidates with explicit empty-state context, while the World premise essence remains refused.

## Decision-Point UI Contract

This flow must satisfy `guided-workflow-usability.md` W-8 and `guided-flow-spec-template.md`.

- **New-world start/resume:** when no `world_kernel` exists, the browser foregrounds Creation as the primary active guided path. World-file controls, Operating Card status, read-only workbench views, and substrate controls may remain available, but unrelated guided flows must show prerequisites or unavailable states instead of competing as equivalent first actions.
- **Kernel authoring:** the browser shows the steward that the decision is to define the world's first governing kernel. Kernel authoring is a composite decision point whose active local unit is the selected kernel section: the selected section's app-owned template prompt or obligation appears beside the field, required sections are distinguished from allowed-empty sections, and the section's expected contribution is stated in app wording with template paths kept as provenance. The surface cites `05` Phase 1 and `templates/world_kernel.md`, requires explicit consequence-mode selection, previews the selected-section kernel prompt packet, records prompt skips, and shows that the resulting write is a living `world_kernel` record rather than admitted canon facts.
- **Prompt-out in context:** Creation Prompt-out appears inside the current decision point. Proposal mode may load for an empty non-premise selected kernel section by carrying the selected heading as explicit request context; pressure mode still waits for steward-authored material in the selected section, and World premise Proposal remains refused by the selected or requested target heading. The prompt-out request is pre-bound to the current flow, step, record, mode, and selected target heading rather than raw record-id entry on the normal Creation path. It previews the prompt packet and source manifest, names omission reasons, warns that pasted responses remain advisory artifacts, and uses the shared Prompt-out lifecycle from ADR 0007.
- **Seed decomposition:** the browser shows that the decision is to split steward material into parked seeds, cites `05` Phase 2 as provenance, and renders the operative Phase 2 rule in app wording: split until each seed can be independently rejected without destroying its siblings, then stop before facts become too small to owe consequences. It shows consumed drafts/kernel material, exposes title/body/truth-layer/granularity controls inside Creation, fixes the actual current seed canon status at `proposed`, keeps any future Admission intent type-separated as advisory intent, previews the decomposition prompt packet, records prompt skips, writes one append-only `seed_decomposition` report, parks seed records at `proposed`, and points the steward to Admission as the next governed step. Server-returned blockers and action failures render at or near the failed action, preserve entered material, and name the remediation path; pre-submit readiness is visible for consequence mode, seed title, seed body, truth layer, and granularity confirmation.
- **Kernel-complete/no-seed map state:** after a kernel exists but before seed decomposition parks proposed seeds, the workflow map remains a Creation orientation surface. It names seed decomposition as the owed current or next decision, shows why proposed seeds unlock Admission, keeps Admission queue `0` from reading as completion, and uses the server-returned Creation resume route rather than a generic review or QA destination.
- **Post-decomposition handoff:** after seed parking, the active Creation surface foregrounds the parked seed short IDs, titles, bodies, truth layers, current `proposed` canon status, seed-decomposition report, derived-from links to the kernel and report, granularity rationale or confirmation, optional Admission intent, Admission queue route, current/next/resume state, and read-side trail. Creation remains clear about the boundary: it parks proposed material; Admission owns first canon standing, frontloaded seed audit, severity, gate results, and canon-status changes.
- **Minimal Viable World checkpoint:** after seed evidence is admitted, the browser foregrounds a Creation checkpoint for phases 4-8. It shows whole-world signals, per-admitted-seed dimension signals, close blockers, current checkpoint report snapshot, unresolved deferrals, open checkpoint debt, Admission proposals, Prompt-out proposal/pressure mode availability, and read-side trail. Recording a disposition requires steward substance; protected mystery dispositions require an existing protected/mystery-boundary record; fact-shaped follow-up routes to Admission as `proposed`.
- **Resume/exit:** the flow map shows current step, kernel/decomposition state, prompt-out status, skipped instruments, and safe resume from `flow_instances`.

## Flow Behavior

- Starting the flow creates or resumes an in-progress `creation` flow instance.
- Kernel authoring creates one `world_kernel` card-regime record if none exists for the flow.
- A saved `world_kernel` without parked proposed seeds leaves Creation active or owed for seed decomposition; it is not downstream canon material for Admission, Propagation, Minimal Viable World, QA, or review foregrounding.
- Each kernel step writes sectioned prose under a seeded heading. Empty sections are allowed.
- The consequence-mode step writes the `consequence_mode` facet only from an explicit steward choice.
- Prompt-out is offered at dependency-bearing steps with mode-specific preconditions. Proposal mode is available for empty non-premise Creation kernel sections when the request carries the selected target heading and explicit empty-state context. Pressure mode remains unavailable for an empty selected section and names the authored-material blocker. World premise Proposal is refused by selected or requested target heading, not merely by the persisted flow step. Decline writes a skip record.
- Kernel Prompt-out uses the selected kernel section as its packet grain when a section is selected. The packet's current decision identifies that heading; decision material and doctrine carry the selected section's prose or empty state and template prompt; proposal mode asks for candidates for that section; pressure mode challenges that section's authored material. A generic kernel packet while a section is selected is insufficient unless the surface explicitly marks a full-kernel pass.
- The selected target heading is request and decision-point context, not stored flow state. Loading Proposal for an empty non-premise section must not auto-save an empty kernel section, advance the Creation flow step, mutate kernel prose, or assign canon standing.
- Decomposition Prompt-out is unavailable or visibly blocked until the seed-decomposition report and parked seed records exist. It must not silently fall back to a kernel-only prompt after decomposition material exists.
- Decomposition consumes selected drafts and/or kernel seed material, writes one append-only `seed_decomposition` report, and parks each seed as a `canon_fact` record with steward-supplied truth layer and actual current canon status `proposed`.
- Parked seeds link to the kernel and decomposition report through app-minted `derived_from` links.
- Completing decomposition marks the flow complete. No record is created with an admitted status by this flow.
- The Minimal Viable World endpoint reads accepted seed `canon_fact` records, preferring seeds linked to Creation kernel/decomposition provenance when available. Proposed-only seeds do not earn the checkpoint.
- Checkpoint coverage signals are advisory clerked evidence, not a verdict. Missing coverage stays visible until the steward records covered, deferred, or protected-mystery dispositions.
- Disposition saves create immutable checkpoint `pass_report` snapshots with `MVW_DISPOSITION` lines, evidence/protected links, skip records, canon debt, and close-readiness state. Later saves supersede earlier snapshots rather than editing report-regime records.
- Checkpoint Admission proposals create `canon_fact` records at `proposed` through Admission intake and link them to the checkpoint report.
- QA consumes the latest checkpoint state as a read-only echo on whole-world passes; QA does not write checkpoint records.

## Out-of-Scope Creation Phases

This spec covers phases 1-2 and the non-generative phases 4-8 Minimal Viable World checkpoint. It still excludes frontloaded seed audit, seed admission, automated world viability judgment, generative ordinary-life establishment, automated path-dependence/faction creation, and iterative canon maintenance. Admission owns first canon standing and the frontloaded seed audit. QA owns the first coherence audit, consuming the checkpoint echo read-only.
