## Recommended First PRD

### Stable Prompt-Out Evidence-List Identity

Purpose: keep Prompt-out omissions and source manifests explicit, complete, and stable when several structural paths produce the same display text.

Sources: Field Build 20 `F-01` and App Seed 1; the exact packet and console evidence summarized by the durable report; current shared Prompt-out and Temporal packet contracts; closed #370 and #380 as adjacent prior art; T-3, W-1, W-8/W-9; and ADRs 0007 and 0009.

Problem: Prompt-out preview collections are string arrays with no identity separate from prose. Temporal graph traversal can encounter the same excluded record through several equivalent second-hop paths and append identical text several times. The browser then uses that text as list identity in both the source manifest and omission list, causing repeated React errors and unstable reconciliation. A browser-only key tweak would suppress the warning without deciding whether the repeated rows are duplicate provenance or distinct evidence.

Recommended product rule or seam:

1. The shared Prompt-out preview contract returns source-manifest and omission items with stable server-owned identity plus display text. Display prose is never the identity.
2. Context assemblers canonicalize items that name the same candidate, exclusion rule, relationship class, and decision meaning. Equivalent graph paths may be summarized with a deterministic path count or provenance set rather than repeated as indistinguishable rows.
3. Items remain separate when their relationship, exclusion reason, standing, or decision meaning is materially distinct. Each separate item carries a unique stable identity and readable provenance.
4. Packet rendering consumes the same canonical item set used by the browser preview. The visible preview, source manifest, omission block, copied packet, downloaded packet, and packet identity remain semantically and byte-contract consistent.
5. Browser renderers use server-returned item identity. Compatibility adapters for older string-only payloads may derive deterministic identity at one boundary, but no component keys directly by display content.
6. Canonicalization and rendering are read-only. They change no record, relationship, flow, revision, packet currentness, advisory state, skip, debt, or canon standing.
7. The Temporal multi-edge graph is the tracer scenario, while shared-contract tests prevent the same failure shape across other Prompt-out surfaces.

Scope:

- Shared Prompt-out evidence-item response and packet-rendering contract for source-manifest and omission collections.
- Temporal context canonicalization for repeated direct and second-hop candidate/exclusion results.
- Shared or migrated browser renderers that currently key Prompt-out evidence rows by display text.
- Exact parity, identity, accessibility, zero-mutation, and console evidence.
- Spec and browser-evidence updates plus focused server, shared renderer, React, and active-route verification.

Acceptance:

1. A real World fixture in which one excluded Temporal record is reachable through four equivalent bounded second-hop paths returns one canonical omission item or one explicitly aggregated item, not four indistinguishable rows.
2. When two omissions share display wording but differ in candidate, exclusion reason, relationship, or decision meaning, both remain present with distinct stable identities and readable provenance.
3. Source-manifest and omission items expose stable identity independent of display text; no Prompt-out browser consumer uses item prose as a React key.
4. Temporal Proposal and Pressure previews render every canonical manifest and omission item once and remain stable across packet load, copy, download, coverage edits, outcome edits, current-packet recovery, finalization, and safe resume.
5. The exact current packet body, copied text, downloaded text, source manifest, omission block, packet hash, and body hash preserve the existing parity contract after canonicalization.
6. No distinct candidate or meaningful provenance path is silently dropped. Aggregated paths expose enough provenance or count information for a cold reader and steward to understand why the context was excluded.
7. Current, incomplete, and stale packet handling remains server-owned; errors and remediation remain adjacent and announced, with selected mode and orientation preserved.
8. Generation, canonicalization, preview, copy, and download are proven read-only through before/after World-file fingerprints.
9. Focused server tests cover equivalent multi-edge duplication, same-text-but-distinct items, deterministic ordering, identity stability, and packet parity.
10. Focused React tests cover identical display values, repeated rerenders, copy/download controls, recovery, accessible list rendering, and a console spy with no duplicate-key errors.
11. A docs-naive production Temporal walkthrough repeats the Field Build 20 packet route and records complete visible omissions, exact export parity, safe return/resume, and zero unexplained console errors or warnings.
12. Root gates `pnpm test`, `pnpm typecheck`, and `pnpm build` pass on the final implementation tree.

Likely issue slices:

1. Shared Prompt-out evidence-item identity and canonicalization spec contract.
2. Server evidence-item model, Temporal multi-edge canonicalization, and packet parity.
3. Browser evidence-list rendering and cross-flow compatibility migration.
4. Active-route Field Build 20 replay and closeout evidence.

Out of scope:

- Changing which world records are eligible Temporal context beyond canonicalizing equivalent results.
- Hiding omissions, merging materially distinct provenance, or weakening incomplete-context blockers.
- Changing Temporal revision, finalization, advisory, skip, or Conditional-pass behavior.
- Reopening #370, #379, #380, or #391.
- Readable pre-start source identity, Constraint Composition, Stage 12 authoring, methodology amendments, or a new ADR.
