## Recommended First PRD

### Pre-Close Temporal Revision Lifecycle

Purpose: let the steward correct Temporal coverage while its run is open, preserve the full audit of what changed, and make close serialize only the final active coverage rather than freezing the first draft.

Sources: Field Build 19 `F-03`/`F-04` and App Seed 2 as summarized here; durable methodology `03`, `09`, and `20`; Temporal, Propagation, schema, Prompt-out, and browser-guidance specs; P-5, P-2/W-1, W-7-W-9, W-5/W-6/T-3/T-5; ADRs 0007-0009; closed #201/#207 and #353/#357/#368; current Temporal source, tests, world-state readback, and refusal log.

Problem: the first coverage save writes directly into an accepted append-only report while the run is still open. Any later correction is rejected, the browser hides the rejection and retains discarded prose, and close remains enabled. Pressure can therefore prove the material wrong without any governed way to repair it before finalization.

Recommended product rule or seam:

1. An open Temporal run owns one active coverage snapshot in flow-owned staging. Coverage is not a pass-report section until close.
2. The first save creates active staging. Every later accepted save creates a new active revision, retires the prior revision as superseded, requires a steward-authored reason when material changes, and preserves prior content, actor, timestamp, flow step, and lineage.
3. The server returns active revision identity, dirty/current state, lineage, validation blockers, close readiness, write intent, and read-side trail. The browser renders those values and does not infer policy.
4. A failed save preserves all entered fields, shows the exact server refusal adjacent to the action in an announced region, and gives a concrete remediation route. The screen must never present rejected text as persisted.
5. Any unsaved or refused revision blocks close. A substantive revision after used Pressure makes the support beat current again through a fresh Pressure packet or the governed Prompt-out skip path; external LLM use remains optional.
6. Successful close writes one append-only final report containing the active coverage and enough revision audit to explain retired versions, then refuses every staging revision action.
7. Existing open runs with a pre-close coverage section are migrated losslessly into active staging without editing that report. Their eventual close writes a new correction report linked with the existing `supersedes` relationship; read-side surfaces identify the corrected final result while retaining the prior report as audit.
8. Source fact standing/text, PRP-1, conditional-pass sibling obligations, Admission contents, canon debt, advisory artifacts, and unrelated world data remain untouched except for explicitly chosen Temporal outcomes at close.

Scope:

- Temporal-flow spec and schema contract for staging, revision lineage, migration, close, and correction-report readback.
- Temporal-owned persistence and HTTP actions for save/revise, active revision reads, close readiness, finalization, post-close refusal, and migrated open runs.
- Shared Prompt-out currentness integration only at the existing lifecycle interface; packet content/reachability remains the #208/#209 conformance action.
- Browser coverage workspace with current/dirty/superseded state, revision reason, preserved drafts, inline failure recovery, finalization landmark, close preview, safe exit/resume, and read-side links.
- Focused tests, existing-world migration proof, one real active-route walkthrough, and the exact Field Build replay.

Acceptance:

1. A new open run can save all ten Temporal lenses, revise at least four lenses with a reason, refresh, exit, and resume with the revised active snapshot and complete retired lineage intact.
2. A second save no longer writes or mutates a pass-report section; the world file contains flow-owned staged versions until close.
3. Server validation failure preserves every editor value and selection, renders the exact error beside the save action, announces it accessibly, and prevents close while the attempted revision is unresolved.
4. Close readiness derives from the active coverage revision, required Temporal substance, explicit outcomes, and current Prompt-out/skip state; superseded coverage cannot satisfy readiness or enter the final active result.
5. Revision changes the Temporal active-set identity and makes packets over the previous coverage stale; recovery offers current Proposal/Pressure generation through the shared lifecycle after the #208/#209 conformance repair.
6. Close preview names the final coverage revision, retained lineage/audit, outcomes and links to be written, and the source/report/queues/debt/advisory state left untouched.
7. Successful close writes one append-only report for a new run, and post-close revision is refused without partial mutation.
8. An existing open run whose first coverage save is already in an immutable report migrates without rewriting that report; corrected close creates a new linked correction report, and read-side/current-result views do not treat the known-defective report as the final result.
9. The Field Build world can reapply the four named corrections, produce the intended timeline card/debt outcomes, continue to the next conditional pass, and retain zero unexplained browser console errors or warnings.
10. Root gates `pnpm test`, `pnpm typecheck`, and `pnpm build` pass on the final implementation tree.

Likely issue slices:

1. Temporal staging/finalization spec, schema, migration, and browser-evidence contract.
2. Server Temporal active coverage revisions, lineage, close readiness, correction-report finalization, and Prompt-out currentness inputs.
3. Browser Temporal revision workspace, preserved-draft failure recovery, close preview, and read-side distinction.
4. Existing-world migration, active-route replay, cold/current packet proof after the conformance repair, and closeout.

Out of scope:

- Rewriting or deleting PAS-1 or any other append-only report.
- A generalized revision framework for every specialized pass.
- Fixing Temporal Prompt-out reachability or packet context as new scope; those are accepted #208/#209 conformance repairs.
- Conditional-pass deferral reinstatement, source-id presentation polish, Constraint Composition or Institutional / Economic / Suppression implementation, Admission changes, direct LLM integration, or methodology amendments.
- Making external LLM use mandatory.
