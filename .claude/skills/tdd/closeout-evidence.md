# TDD Closeout Evidence

Read this before handing TDD work to review, drafting a pre-close audit, closing issues or a parent PRD, or mutating the tracker. This file is the single source of truth for validator use, interim and final durable-sink gates, evidence freshness, and the parent-rollup-ready template.

## Validator and Non-Bypassable Gate

When the TDD evidence is being drafted in a file, run `node .claude/skills/tdd/scripts/validate-tdd-closeout-body.mjs <body.md>` before review or a pre-close audit. When the exact body will be posted or linked for tracker closeout, run `node .claude/skills/tdd/scripts/validate-tdd-closeout-body.mjs <body.md> --closing --expected-final-sha "$(git rev-parse HEAD)"`; closing mode requires that live-HEAD value, compares it with every `Final SHA:` and `reviewed HEAD SHA` claim, rejects absolute local paths in publishable sink fields while still allowing fixture and artifact paths elsewhere, requires the final-tree verification command ledger below, and enforces a 65,536-byte body maximum by default. `--max-bytes <positive integer>` may lower that ceiling or match a different tracker's documented contract; it must not be used to bypass GitHub's limit. Add `--parent-rollup` when the evidence uses the parent-rollup compact table in a parent PRD rollup, including a single child plus parent PRD closeout and small child families that choose a parent rollup as the durable sink. Parent-rollup, parent-PRD, child-family, and named issue-set validation also requires `--acceptance-manifest <path>` generated from saved exact issue JSON; the validator checks every generated parent section, individual `USN`, AC, and Principles ID against the compact rows and adjacent keyed maps. AC ranges may enumerate issue-qualified AC checks, but a `US1-US36` range does not replace individual story rows. If the evidence exists only in conversation or a tracker UI, apply the same checks manually and say the validator was N/A because no local body file existed.

When `implement` owns a combined parent/issue closeout, plan size before filling TDD evidence:

```bash
node .claude/skills/implement/scripts/build-closeout-body.mjs <manifest.json> --audit-input <audit.md> --output <body.md> --parent <issue> --review <normal|fallback> --tdd-parent-rollup --size-plan --require-headroom
```

Carry the other applicable implement flags into that command. A `low-headroom` or `exceeds-limit` result is a precomposition stop, not permission to shorten atoms, proof surfaces, or sequence evidence. Generate disjoint subset manifests and matching audit scaffolds with repeatable `--select <issue[:check-id[,check-id...]]>` arguments to `.claude/skills/implement/scripts/build-acceptance-manifest.mjs`, then allocate the corresponding TDD issue/seam rows to separately validated chunks. Ensure the subsets are disjoint and collectively cover the full authoritative manifest.

Before tracker mutation, run `wc -c <body.md>` as an independent final size readback. If the exact body still exceeds the provider ceiling, do not weaken atoms or proof surfaces to recover space. Split the detailed TDD table into the planned disjoint issue/seam groups, validate each chunk with the matching subset acceptance manifest and appropriate flags, post and read back every chunk, and cite all stable comment URLs from the compact closing body and affected child comments. Validate the compact body separately against only the rows it contains; do not claim one validator invocation covered rows stored in other comments.

Non-bypassable TDD closeout gate:

- Before handing work to review, inspect an interim durable audit sink that preserves TDD evidence for review. This can be the conversation, implementation ledger, or a body draft if the final parent rollup or closeout body cannot exist yet.
- Before a pre-close audit, issue closure, parent PRD closeout, or the repo `implement` skill's closeout gate, inspect the final durable closeout sink or body that will be posted or linked.
- A private pre-post scratchpad or interim validator run may name an inspected local body file before the tracker URL exists. The body sent to the tracker must instead use a comment URL, stable issue reference, or self-referential description that is true before and after posting; never publish the local staging path or `pending tracker URL`. If either is posted by mistake, patch the tracker comment before issue or parent closure.
- Both the interim and final durable sinks must contain the compact table below or equivalent explicit fields: issue(s), `CONTEXT.md` status, ADRs/principles/docs status, seam, red command/failure or skip reason, green command/evidence, and acceptance covered with the exact criterion, authoritative atoms, required proof surfaces, and ordered proof or a justified sequence N/A.
- Structural table check: report `compact table/header present` only when the exact compact TDD table header is present in the sink. Report `equivalent fields present` only when every in-scope issue/seam entry explicitly carries the same minimum fields: issue(s), `CONTEXT.md` status, ADRs/principles/docs status, seam, red command/failure or skip reason, green command/evidence, and acceptance covered with `atoms:`, `proof surfaces:`, and `sequence:`. Keyword hits alone do not satisfy this check.
- Row-content check: every `Red command/failure` cell must contain an exact executable command in backticks plus its failure signal, an explicit `N/A because ...` or `red-first skipped because ...` reason, `partial red - wrong reason: ...`, `coverage-only review fix ...`, `coverage-only existing behavior; red-first N/A because behavior already existed and no production code changed`, `existing contract-change expectation in <test file> because ...`, or a named shared-red-command reference. Backticked identifiers, fields, test names, and file paths are not commands. Summaries such as "covered by lifecycle assertions" or "initial tests failed before the feature existed" are not red evidence unless they cite the exact failing command or linked shared command.
- Final-tree verification command ledger: every closing body must include the exact header `| Exact command | Observed result/counts | Run count | Represented SHA/tree |` and at least one row. Put the full executable command in backticks, copy an output-derived result or count such as `passed: 2 files and 5 tests; exit 0`, record how many times that exact invocation ran, and name the commit SHA represented by the run. A prose claim such as `focused tests passed`, an intended command that was not the final invocation, or a row representing an earlier tree is not durable verification. Every row must represent the body's `Final SHA:`; rerun or mark the closeout blocked when a later edit makes a row stale.
- Acceptance-atom and sequence exactness check: every compact row's `Acceptance covered` cell must cite the exact acceptance criterion or critical clause, then include `atoms:`, `proof surfaces:`, and `sequence:`. Resolve composite terms through their authoritative parent/child definition, implementation decision, glossary, or named contract; name every required atom and map it to every promised surface. Do not point back circularly with wording such as `every exact named contract in the issue criteria`, `exact named items in this criterion`, `the criterion above`, or `all named items`. Every proof surface must have a concrete anchor in the same row: a test or document path, verification command, API/browser route, artifact, URL, tracker reference, or a concrete `Green command or evidence` value that names the surface and observed result. A quantified range such as “two through five” requires proof at every named value unless the source explicitly permits boundary or representative sampling. For order-, transition-, or timeline-sensitive criteria, name the ordered events and the proof that observes their order. A lifecycle term such as rerender, resume, or transition requires one active instance plus the triggering action and observer that prove continuity; independent snapshots or server renders are not equivalent. Otherwise use `sequence: N/A because the criterion is not sequence-sensitive`. Use `atoms: atomic` only when the authoritative criterion has one indivisible obligation. A broad umbrella noun such as `provenance` does not satisfy the row when the authority requires `actor + timestamp + flow step` across `API + report + browser`, and independently observed states do not prove a required sequence.
- Evidence-only exactness check: for evidence-only browser/manual/report/external rows, `Acceptance covered` must name the exact acceptance criterion or critical clause that the evidence proves and may additionally cite a per-criterion audit row in the same durable sink for the detailed proof. The compact row still carries its own `atoms:`, `proof surfaces:`, and `sequence:` map. Broad labels such as `active-route replay`, `workflow coverage`, `browser evidence`, or `cold proof` are not enough when the source criterion requires a specific condition like same-runtime switching, exact routed surface, selected record, a named stale/current exclusion, an ordered walkthrough, or an exact external/cold probe result.
- Evidence-only external/cold proof check: if a TDD row claims cold external LLM, fresh subagent, packet-read, nonlocal probe, or similar external evidence, the row must either carry that exact proof in `Green command or evidence` or cite the same durable sink's per-criterion audit row that names the proof. A focused test command alone does not cover cold/external acceptance.
- Evidence-only browser/manual console check: for evidence-only browser/manual rows, record console error/warning state in the durable sink. If the proof spans hot reload/HMR, reused browser sessions, agent-induced setup/request errors, or unexplained console errors or warnings, treat that row as preliminary until a clean browser session reruns on the final tree or the console output is classified as unrelated with evidence.
- Evidence-only proof-server preflight check: before starting or attaching to a proof server, record `Evidence-only proof server preflight:` with the configured API/UI ports and owner-check result, unrelated pre-existing owners or `none`, the configured-ports-free result or isolated proof-owned ports plus aligned proxy/API base, and cleanup ownership. Use a reasoned N/A only when there are no browser/manual evidence-only rows or no proof server applies; never stop or reuse an unrelated pre-existing process as proof.
- Evidence-only backend-currentness check: when browser/manual proof consumes a backend, record `Evidence-only backend process currentness:` with the server command and watch/reload mode, process or port ownership from the proof's execution context, restart/reload proof after relevant backend changes, and the expected API field/behavior probe run before the final UI assertion. When that proof uses a copied stateful fixture, the same field also names the snapshot method and source plus the expected-state probe; for Worldloom SQLite with possible live WAL state, use `.backup` or an explicit checkpoint-aware copy rather than raw `cp`. Reachability and a file that merely opens are not currentness. Use a reasoned N/A only when the browser proof has no backend/API dependency, no stateful fixture was copied, or there are no browser/manual evidence-only rows.
- Four-column summary tables are not compact TDD evidence. A table like `| Issue | Seam | Red-first status | Green verification |` is a summary only; it cannot justify `compact table/header present`, and it is not `equivalent fields present` unless every row separately carries the missing `CONTEXT.md` status, ADRs/principles/docs status, red command/failure or skip reason, and acceptance-covered fields. When a parent PRD rollup carries child-family TDD evidence, use the parent-rollup-ready eight-column table below in the final rollup or link an adjacent durable sink that contains it.
- Make the literal `TDD evidence gate passed:` line visible in the conversation, implementation ledger, or durable audit sink. If any field is missing or only implied, stop and add it before leaving the TDD loop.
- Do not write or accept the shorthand `TDD evidence gate passed: yes`. The gate line must be the full fielded sentence shown below, including `durable sink`, `compact table/header`, `seams accounted for`, `CONTEXT.md status`, `ADRs/principles/docs status`, sequence evidence, evidence identities, partial-red or skip-reason status, evidence-only row status, proof-server preflight status, and existing-test contract-change row status. Do not write `compact table/header present` unless the structural table check above is true.
- Pre-red recovery check: the closeout preflight must record whether the pre-red preflight/table was visible before the first red command or list the `TDD recovery addendum:` that preserved reconstructed evidence after a missed implement-owned pre-red gate. If the implement-owned pre-red gate was missed and no recovery addendum is preserved in the durable sink or an explicitly linked adjacent sink, stop and add it before review, pre-close audit, tracker mutation, or implementation closeout.
- Evidence-identity reconciliation check: the final sink must contain one shared `Evidence identity refresh:` block with `Current evidence identities:`, `Historical red identities retained:`, `Superseded evidence identities:`, and `Superseded-token sweep:`. List fixture paths, browser sessions, packet paths/hashes, active revisions, and artifacts under the current and superseded fields. When authority forbids publishing a local fixture path, use `fixture paths withheld because <authority and reason>; logical fixture <stable ID>; content SHA-256 <64 hexadecimal characters>; provenance <generated, derived, or copied-source statement>`; never use `none published because`. Historical red identities may remain when they are explicitly classified as red history rather than current proof. When `implement` owns closeout, reuse its same identity block and add the historical-red field instead of creating a second inventory.
- Nested-validator angle-token check: in compact TDD, review, or audit cells and in evidence-identity values, avoid HTML-like angle tokens such as a backticked body tag even when the token is intentional. Spell it in prose, for example `document body`, because a nested validator may classify the whole value as an unresolved placeholder.
- Evidence-artifact lifecycle check: before deleting proof-owned fixtures, scripts, packets, or artifacts, compare them with the published `Current evidence identities:` inventory and closeout references. A published current artifact is not safe to remove until closeout is complete and its retained-or-removed disposition is recorded. If it is removed after publication, update the durable evidence so it does not imply continued local inspectability, while retaining the command/result or tracker-hosted evidence that supports the claim.
- Body-check token sweep: before declaring the gate satisfied, grep or visually verify that the durable sink contains `TDD closeout preflight`, `TDD evidence gate passed: durable sink`, `compact table/header`, `seams accounted for`, `CONTEXT.md status`, `ADRs/principles/docs status`, `Acceptance atom map:`, `Acceptance sequence map:`, `partial-red / red-first skip reasons`, `evidence-only rows`, `Pre-red recovery status:`, `Evidence-only proof server preflight:`, `Evidence-only backend process currentness:`, `Evidence identity refresh:`, `Current evidence identities:`, `Historical red identities retained:`, `Superseded evidence identities:`, `Superseded-token sweep:`, and `Existing-test contract-change rows:`; when evidence-only browser/manual rows exist, also verify `Evidence-only browser console state:`. Then perform the structural table check and `wc -c` size readback. The token sweep is a locating aid, not proof by itself. When a local body file exists, run the validator command above and fix any reported structural error before declaring the gate satisfied.
- When the same durable body also carries local `code-review` fallback evidence, include the code-review fallback `TDD closeout gate:` field in that body. The field must point to or include the inspected `TDD closeout preflight` block and full `TDD evidence gate passed:` line. When the sink is a local file, run `node .claude/skills/code-review/scripts/validate-review-fallback-body.mjs <body.md> --implement --tdd` before handoff, using `--tdd-parent-rollup --acceptance-manifest <path>` instead of `--tdd` when TDD evidence uses the parent-rollup compact table, adding `--child-family --acceptance-manifest <path>` when the review fallback covers a child-issue family, and adding `--browser` when browser/manual evidence is present.
- When `implement` owns the surrounding workflow, copy or link the interim TDD evidence into the same final parent rollup, issue comment, or body file that carries implementation closeout evidence before the pre-close audit, issue closure, or parent PRD closeout. A verification section that lists red/green commands is not equivalent unless it also names the issue/seam rows, `CONTEXT.md` status, ADRs/principles/docs status, skip or partial-red reasons, and evidence-only freshness.

Tracker mutation hard stop: before any issue closure, parent PRD closeout, or the repo `implement` skill's closeout gate, inspect the final body that will be posted or linked and run the TDD validator with `--closing --expected-final-sha "$(git rev-parse HEAD)"` when a local body exists. Stop unless it passes live-SHA closing validation, the final-tree verification command ledger, the 65,536-byte default ceiling, and the structural table check and contains the exact full `TDD evidence gate passed: durable sink ...` line with `CONTEXT.md status`, `ADRs/principles/docs status`, sequence evidence, evidence identities, partial-red / red-first skip reasons, evidence-only rows, proof server preflight, and existing-test contract-change rows.

## Review-Fix Evidence and Freshness

- **TDD review-fix addendum.** When review creates a TDD row or changes an existing one, paste this compact addendum into the implementation ledger, review fallback block, or final closeout sink. Equivalent fields are acceptable only when the same durable sink explicitly contains the seven base field labels below (`Finding:`, `Intended red command/failure:`, `Green command/evidence:`, `Updated TDD table row:`, `Browser/manual evidence freshness:`, `Backend process currentness:`, and `Evidence identity refresh:`) and links or names the updated compact-table row. The validator accepts the legacy `Browser/manual freshness:` alias in existing bodies, but new bodies use the shared canonical label. When the intended red is a transient browser/manual probe, `Regression durability:` is also mandatory.

  ```markdown
  TDD review-fix addendum:
  - Finding: <review finding title/source>
  - Intended red command/failure: <command and failure / partial red - wrong reason plus follow-up intended red / red-first skipped because ... / red-first skipped because Standards-only/conformance-only fix did not change behavior / coverage-only review fix because ...>
  - Green command/evidence: <passing command / browser/manual evidence / artifact>
  - Updated TDD table row: <issue/seam row updated or linked>
  - Regression durability: <durable test added at path / evidence-only because no supported committed harness exists / blocked because ... / N/A because the intended red was not a transient browser/manual probe>
  - Browser/manual evidence freshness: <rerun result / not affected because changed path <path> is outside the evidence-only route/action/API/fixture <route/action/API/fixture> and targeted proof <command> passed / not affected because changed path <path> is non-semantic formatting, comment wording, documentation, or closeout text; evidence route/action/API/fixture <route/action/API/fixture> is untouched; diff inspected and targeted proof <command> passed / N/A because no UI/routes/browser-consumed API/fixtures/action path changed / blocked because ...>
  - Backend process currentness: <server command and watch/reload mode; process or port ownership; restart/reload proof; expected API field/behavior probe; copied stateful fixture snapshot method/source and expected-state probe or N/A because no stateful fixture was copied / N/A because browser proof has no backend/API dependency / N/A because no browser/manual proof was used / blocked because ...>
  - Evidence identity refresh: <same-sink current/historical-red/superseded identity block inspected / blocked because ...>
  ```

- **Evidence freshness after review fixes.** After review fixes or any later edit, compare the final changed files against evidence-only browser/manual rows before closeout. If a changed file touches a route, UI action, browser-consumed API shape, fixture, or data setup covered by one of those rows, rerun that evidence on the final tree. If a behavior-changing UI edit is outside the evidence-only route/action path, record `Evidence freshness: not affected because ...` only when you name the changed path, explain why the evidence-only route/action path and browser-consumed API/fixtures were untouched, and rerun targeted proof for the changed path. Record `Evidence freshness: <rerun command/artifact/result>`, `Evidence freshness: not affected because ...`, or `Evidence freshness: N/A because ...`; stale or blocked evidence must be explicit. For an active browser/manual rerun on the final tree, use validator-safe wording such as `Evidence freshness: browser smoke rerun passed on final tree for route/action/API/fixture <route/action/API/fixture> with observed outcome <outcome>`. In the closeout preflight, mirror that as `Evidence-only rows freshness: browser smoke rerun passed on final tree for route/action/API/fixture <route/action/API/fixture> with observed outcome <outcome>`.
- **Backend process currentness after review fixes.** For an active browser/manual rerun that consumes a backend, fill both `Backend process currentness:` in the review-fix addendum and `Evidence-only backend process currentness:` in the closeout preflight. Name the server command and watch/reload mode, process or port ownership, restart/reload proof after relevant server/API edits, and the expected API field/behavior probe. When a copied stateful fixture is involved, also name its snapshot method/source and expected-state probe, or state that no stateful fixture was copied. If a stale process or fixture caused an error or currentness is uncertain, restart the proof-owned backend, rebuild the fixture consistently, and rerun in a clean browser session before claiming fresh evidence.
- **Evidence identity refresh after review fixes.** Refresh the shared identity block after every review fix or evidence rerun. Current and superseded fields must enumerate `fixture paths`, `browser sessions`, `packet paths/hashes`, `active revisions`, and `artifacts`; `Historical red identities retained:` must name any earlier red-only identifiers that intentionally remain. The superseded-token sweep names every exact superseded value and the `rg`/`grep` command, then reports `no hits outside classified identity/history lines and no active-proof hits`, followed by any classified historical-red hits.
- If the later edit is non-semantic formatting, comment wording, documentation, or closeout text, inspect the diff and record `Evidence freshness: not affected because changed path <path> is <non-semantic reason>; evidence route/action/API/fixture <route/action/API/fixture> is untouched; targeted proof <command> passed` instead of rerunning browser/manual evidence.
- If the final browser/manual proof ran on the same tracked file content that was later committed, do not force a fake rerun only to attach a SHA. Record `Evidence freshness: not affected because git commit metadata only; no tracked file content changed after the proof; evidence route/action/API/fixture <route/action/API/fixture> is untouched; targeted proof git diff HEAD -- <owned files> passed or was empty`.

## Implementation Closeout and Parent Rollup

- **Implementation closeout consumes the TDD table.** When the repo `implement` skill owns the surrounding workflow, preserve the compact TDD evidence table in the implementation ledger or another interim durable sink before review, then carry the same fields into the pre-close audit or final closeout sink before tracker mutation. Immediately before the pre-close audit, either post a short `TDD evidence` block or explicitly carry the TDD fields into the audit evidence. Put that table or block in the same durable sink as the pre-close audit, or directly adjacent to it with an explicit link or citation; do not rely on earlier prose-only red/green notes surviving context compaction. The minimum evidence is: issue or issues, `CONTEXT.md` status or `N/A`, ADRs/principles/docs status or `N/A`, seam, red command and expected failure or skip reason, green command or evidence, and acceptance covered with atoms, proof surfaces, and sequence. If a seam has no row, add the explicit red-first skip reason before leaving the TDD loop.
- **TDD evidence artifact stop.** Before the pre-close audit, issue closure, or parent PRD closeout, inspect the actual durable closeout body that will be posted or linked. It satisfies this skill only when it contains the compact table header row below, or when every entry explicitly carries the same minimum fields: issue(s), `CONTEXT.md` status, ADRs/principles/docs status, seam, red command/failure or skip reason, green command/evidence, and acceptance covered with `atoms:`, `proof surfaces:`, and `sequence:`. Prose-only red/green bullet lists do not satisfy this stop, even if the commands are present.
- **TDD closeout preflight.** Before handing off to review, fill and paste this block in the conversation, implementation ledger, or other interim durable sink. Before a pre-close audit, issue closure, or parent PRD closeout, paste or update it in the final closeout audit or body. This block is only the checklist for the gate; it is not the compact evidence table. A prose sentence headed `TDD closeout preflight:` is not enough: the durable sink must carry the field labels below. Do not proceed until the compact table or equivalent per-row fields are adjacent or explicitly linked, and the `TDD evidence gate passed:` line is present immediately after the preflight and shared identity block.

  ```markdown
  TDD closeout preflight:
  - Durable sink/body inspected: <conversation/comment URL/stable issue reference before tracker URL exists>
  - Compact table/header: <present after structural check/equivalent fields present after structural check/missing>
  - Rows accounted for: <all in-scope issues and seams listed / exceptions named>
  - Pre-red recovery status: <N/A - pre-red preflight/table was visible before first red / listed with TDD recovery addendum / N/A because no red commands were run / blocked because ...>
  - CONTEXT.md status: <present/absent/N/A>
  - ADRs/principles/docs status: <present/N/A>
  - Acceptance atom map: <all rows list exact criterion plus authoritative atoms and proof surfaces / all criteria atomic and rows list proof surfaces / blocked because ...>
  - Acceptance sequence map: <all rows list ordered proof, with rerender/resume/transition continuity observed on one active instance, or justified sequence N/A / blocked because ...>
  - Partial-red / red-first skip reasons: <none/listed>
  - Evidence-only rows freshness: <none/listed with Evidence freshness rerun/not affected/blocked result, including non-semantic formatting/comment/docs-only or commit-metadata-only freshness when applicable>
  - Evidence-only browser console state: <none / listed with 0 errors and 0 warnings, classified unrelated output with evidence, rerun clean session because HMR, reused session, or agent-induced setup/request error tainted proof, or N/A because no browser/manual evidence-only rows>
  - Evidence-only proof server preflight: <configured API/UI ports and owner-check result; unrelated pre-existing owners none/listed; configured ports verified free or isolated proof-owned ports plus aligned proxy/API base; cleanup ownership / N/A because no browser/manual evidence-only rows or no proof server applies / blocked because ...>
  - Evidence-only backend process currentness: <server command and watch/reload mode; process or port ownership; restart/reload proof; expected API field/behavior probe; copied stateful fixture snapshot method/source and expected-state probe or N/A because no stateful fixture was copied / N/A because browser proof has no backend/API dependency / N/A because no browser/manual evidence-only rows / blocked because ...>
  - Evidence identity refresh: <same-sink current/historical-red/superseded identity block inspected / blocked because ...>
  - Existing-test contract-change rows: <none / listed expectation-rewrite rows with failing command and green rerun>
  ```

  ```markdown
  Evidence identity refresh:
  - Current evidence identities: fixture paths <paths/none/structured withheld identity>; browser sessions <names/none>; packet paths/hashes <paths and hashes/none>; active revisions <IDs/none>; artifacts <paths/IDs/none>
  - Historical red identities retained: <identities and exact red rows where they remain / none>
  - Superseded evidence identities: fixture paths <paths/none>; browser sessions <names/none>; packet paths/hashes <paths and hashes/none>; active revisions <IDs/none>; artifacts <paths/IDs/none>
  - Superseded-token sweep: <`rg`/`grep` command naming every exact superseded value; no hits outside classified identity/history lines and no active-proof hits; historical-red hits classified or none / N/A because every superseded category is none>
  ```

- **TDD evidence gate line.** After inspecting the current interim sink or final closeout body and before leaving the TDD loop for review or closeout, write this literal line in the conversation, implementation ledger, or closeout audit:

  `TDD evidence gate passed: durable sink <conversation/comment URL/stable issue reference before tracker URL exists>; compact table/header <present after structural check/equivalent fields present after structural check>; seams accounted for <all listed / exceptions named>; CONTEXT.md status <present/absent/N/A>; ADRs/principles/docs status <present/N/A>; sequence evidence <present/N/A>; evidence identities <present/N/A>; partial-red / red-first skip reasons <none/listed>; evidence-only rows <none/listed>; proof server preflight <present/N/A>; existing-test contract-change rows <none / listed expectation-rewrite rows>.`

- **Literal gate validation.** Before reporting the TDD evidence gate as passed, inspect the exact durable sink or body and verify the literal line above is present with these field labels: `durable sink`, `compact table/header`, `seams accounted for`, `CONTEXT.md status`, `ADRs/principles/docs status`, `sequence evidence`, `evidence identities`, `partial-red / red-first skip reasons`, `evidence-only rows`, `proof server preflight`, and `existing-test contract-change rows`. A paraphrase such as "includes red-first evidence, focused green evidence, and browser evidence" is not the TDD evidence gate. If the compact table header is absent, the line must say `equivalent fields present after structural check` only after checking every row carries the minimum fields.

- **Existing-test contract-change row exact wording.** When a row uses `Seam` = `existing contract-change expectation`, the `Red command/failure` cell must start with the unhyphenated phrase `existing contract-change expectation in <test file> because ...`, followed by the failing existing-test command/assertion. Do not put `existing-test contract-change expectation` in the red cell; that phrase belongs only in closeout labels such as `Existing-test contract-change rows:` or the review-fix N/A explanation.

- **Parent-rollup-ready TDD evidence fragment.** When a parent PRD rollup carries child-family TDD evidence, including a single child plus parent PRD closeout, paste or adapt this block into the final rollup, or link an adjacent durable sink that contains it:

If the parent PRD itself is in scope and child rows own the runnable seams, include a parent mapping row rather than silently omitting the parent. The parent row can use `N/A because parent PRD maps already-tested child seams rather than adding a new behavior seam` in `Red command/failure`, with `Green command or evidence` pointing to the child rows and root gates.

  ```markdown
  TDD evidence

  | Issue | CONTEXT.md status | ADRs/principles/docs status | Seam | Red command/failure | Green command or evidence | Acceptance covered | Review fix / red-first skip reason |
  |---|---|---|---|---|---|---|---|
  | #N | <read / absent / N/A> | <ADRs/principles/docs read or N/A> | <red-first / existing contract-change expectation / static contract check + browser evidence / coverage-only existing behavior / no-runnable / evidence-only seam> | <command plus expected failure / partial red - wrong reason / coverage-only existing behavior; red-first N/A because behavior already existed and no production code changed / N/A / explicit skip reason> | <passing command / browser route-action-result / external/cold proof artifact or audit-row citation / artifact path / N/A> | <exact criterion or checkbox; atoms: authoritative atoms or atomic, including every named range value; proof surfaces: surface for each atom; sequence: ordered proof plus same-instance action and observer / N/A because criterion is not sequence-sensitive> | <N/A / review-fix evidence / partial-red follow-up / red-first skip reason> |

  Existing-test contract-change rows: <none / listed expectation-rewrite rows in the table above with the failing existing-test command and the focused or broader green rerun>
  Example existing-test row:
  ```
