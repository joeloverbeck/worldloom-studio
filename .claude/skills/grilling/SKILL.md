---
name: grilling
description: Interview the user relentlessly about a plan or design. Use when the user wants to stress-test a plan before building, or uses any 'grill' trigger phrases.
---

Interview me relentlessly about every aspect of this plan until we reach a shared understanding. Walk down each branch of the design tree, resolving dependencies between decisions one-by-one. For each question, provide your recommended answer.

Before asking, classify the request. If it is a design or plan stress-test, run the full interview. If it is an operational, triage, dependency-maintenance, or other delegated-execution request that uses grilling to sharpen judgment, explore the live facts first, resolve only the decision branches that materially affect the action, keep the ledger minimal, and proceed with the explicitly requested work once those branches are resolved. When the requested action may edit files, include a worktree check such as `git status --short` before edits and distinguish pre-existing/user artifacts from intended outputs. When the action instead mutates a tracker, remote, or other external surface rather than repo files, capture the equivalent baseline — the affected issue/PR numbers, labels, and dependency lines — before the first write, re-read it before the final summary, and state explicitly when the repo worktree is untouched. When the change is to a versioned repo file but is committed directly on the remote — a server-side commit or contents-API write, for example because the local branch has diverged — the local worktree checklist is N/A: capture the remote file's pre-write identity (path, blob sha, and branch), state that the local worktree is untouched, and prove the end state with a remote read-back of the committed content. Do not turn an operational request into a broad design interview unless the user asks for that.

For operational requests, refresh drift-prone live facts before freezing the verdict whenever practical, or explicitly label the snapshot limitation. Examples include remote Git refs, GitHub issue/PR state, CI status, package or dependency registry state, deployment state, and runtime process state. Keep these refreshes bounded: prefer exact reads with projected fields, such as `gh issue view <n>` with a `--jq` projection, over broad `--search` or `--state all` listings that dump many irrelevant rows and blow the tool-result budget. If refresh is blocked by permissions, network access, or tool limits, record that block in the evidence and keep the recommendation scoped to the verified snapshot.

For high-risk operational actions, add a hard checkpoint before execution. High-risk actions include ref rewrites or resets, branch deletion, force push, broad file deletion, irreversible data migration, and any command that can orphan or discard local state. They also include outward-facing actions that are effectively irreversible once seen — creating or editing tracker issues and pull requests, merging a pull request into a shared or default branch, posting comments, changing labels, pushing branches, and committing or pushing to the default or protected branch — because a published issue can be closed but never unpublished. Before acting, verify the clean or intentionally-dirty starting state, check for in-progress operations when relevant, create or name a rollback/backup path, state the exact command(s) — or, for outward-facing writes, the exact issues, edits, and labels — plus the expected end state, and get explicit confirmation. An explicit up-front request grants permission but does not satisfy this pre-flight enumeration: the user must be able to see what will be published or changed before it happens. After acting, verify the resulting state with the smallest truthful proof surface, such as status, graph, diff, health check, tracker read-back, or command output.

Classification quick-reference — identify the class first, then work its minimum design tree (the paragraphs above and below elaborate each; a class can shift mid-session as facts surface):

- **Design / plan stress-test** — a plan or design already exists to pressure-test: run the full interview down that plan's own design tree.
- **Determination / recommendation** — choose or generate a plan that does not yet exist: minimum tree is candidate choice, scope boundary, and deliverable depth; then grill the winning candidate's design tree one branch at a time.
- **Diagnostic / audit** — assess, compare, or diagnose something that already exists (has X diverged? why? what to do?): candidate choice is N/A; minimum tree is scope boundary and deliverable depth — a floor, not a ceiling, since a finding that exposes a divergence between the user's stated intent and repo authority re-opens the tree as a ratification branch.
- **Operational / triage / delegated-execution** — explore the live facts first, resolve only the branches that materially affect the action, keep the ledger minimal, and proceed with the explicitly requested work.

Report-cited PRD-ready prep quick path: when a report-cited determination asks to get ready for a later `/to-prd`, work the quick path in `references/prd-ready-determination-artifact.md` §Quick Path For Report-Cited Prep, then write the artifact using that file's skeleton. Ask exactly one artifact-home question, naming the selected scope and target path and including a "recap only, no file" option. When the artifact will include likely label, downgrade conditions, or browser-visible guidance mapping, read the repo's issue-tracker and triage-label docs named by entrypoint guidance before drafting those fields. Classify both the source report and the authored artifact as durable, dirty, untracked, or temp-only; summarize temp or unpublished evidence instead of citing it as stable. Consult `/to-prd` for house style only.

For design stress-tests that cite a report, module, architecture review, artifact, or repo authority, do a minimal intake before the first question: open the cited artifact, read the relevant glossary/principles/ADRs, inspect the current implementation surface, and turn only unresolved stewardship or design choices into questions. When the cited candidate is explicitly about module depth, seams, interface shape, or architecture vocabulary, treat the repo's relevant design-vocabulary or architecture skill as another authority to read during this intake. When the invocation itself names a supporting skill — for example a wrapper such as `grill-with-docs` that runs grilling "using the `/domain-modeling` skill" — load that named skill during intake so its judgment shapes the branches and its result has a recap home. If the repo declares a tracker, backlog, PRD, issue, or run-state surface, and the cited candidate is marked in-process, names prior PRD-shaped work, or otherwise looks likely to overlap live work, run a narrow tracker overlap check against open and relevant closed items; record the result as an explored fact rather than asking the user to remember it. Keep this intake just deep enough to avoid asking what repo truth already answers. When a large report contains multiple candidates or sections, inspect the selected candidate section first and broaden only when dependencies, contradictions, or missing context require it. When a report-cited determination depends on old field-build, audit, review, or PRD evidence, reconcile the report's date, commit, and freshness claims against current source plus relevant open and closed tracker work before assigning PRD scope. If live source or closed work contradicts a reported finding, classify that item as covered unless replay disproves it, or as a verification/reopen candidate, rather than treating it as fresh product scope.

When a report-cited reassessment discovers an existing PRD-ready, issue-ready, or prep artifact for the same source, classify that artifact explicitly as current, partially consumed, stale, superseded, or not relevant. State whether the run updated it, left it untouched, or recommends a follow-up refresh; if one candidate from the artifact was already published or implemented, name the consumed tracker IDs and preserve any remaining candidate separately rather than treating the artifact as wholly current.

When the request is instead to *determine, choose, or generate* a plan rather than stress-test one that already exists (for example, "decide the next PRD"), there is no plan yet to grill: intake is legitimately broader — survey the surface to assemble the candidate set — and the root question becomes the choice among those candidates, after which you grill the winning candidate's design tree one branch at a time. When the user explicitly names a large document glob or asks to "read carefully" a broad authority set, either perform the exhaustive read if feasible or state a bounded intake strategy before relying on it: prioritize indexes, coverage ledgers, tracker state, current candidate files, and authority-order documents; use targeted scans for lower-priority package files; and do not imply line-by-line coverage for files only scanned. If the repo declares a tracker, backlog, PRD, issue, or run-state surface, include the current open items and latest relevant closed parent/PRD in that survey so the candidate set does not duplicate or contradict live work. For determination/recommendation runs, the minimum design tree is usually candidate choice, scope boundary, and deliverable depth; once those are resolved, stop at a PRD-ready or issue-ready recap unless the user explicitly requested publication, issue creation, document writing, or implementation. Preparatory phrasing that asks you to ready the plan for a defined next step — for example "get everything ready to create PRDs", "prep the plan", or "set this up so we can create issues" — counts as an explicit request for a written PRD-ready determination artifact, not merely a recap; it satisfies the "explicit request" bar the deliverable rules below apply to document writing. In that case you may ask exactly one artifact-home question that includes a "recap only, no file" option — that single question is the sanctioned resolution, not the forbidden second artifact-specific question. If the question tool is unavailable and prose fallback is used, preserve the same choice explicitly: `Question N: artifact home. My recommendation (Recommended): write <path> because <reason>. Alternative: recap only, no file. Do you agree?` When that artifact-home prompt names the artifact path and scope, and the user clearly says to write it or proceed, treat that answer as satisfying the downstream-artifact checkpoint unless new user-owned scope decisions have surfaced. PRD-ready prep fast path: if the artifact-home prompt already names the selected scope and target path, that confirmation is the terminal downstream-artifact checkpoint; if the scope boundary was confirmed in a prior question, ask only the artifact-home question next and treat that answer as the terminal checkpoint unless new user-owned scope appears. This is the same determination/recommendation case the deliverable rule below already handles; the recap still carries the choice as its determination. Candidate choice is user-owned unless repo facts conclusively leave only one valid option: record conclusive supporting facts as RATIFIED, but keep a recommended winner PROVISIONAL or outside the decision ledger until the user confirms it. For issue-ready recaps, after the relevant decisions are ratified, you may read an issue-slicing skill such as `/to-issues` for house-style and granularity checks; using it this way is not publication, and tracker issue creation still requires an explicit user request. When the ratified deliverable includes creating PRD or issue artifacts in-session, you may likewise read the repo's PRD-shaping skill such as `/to-prd` — or mirror the repo's most recent ratified PRD or issue — for house style; consulting house style this way is not extra publication scope. For prep-only requests that explicitly name a later `/to-prd` pass, consult `/to-prd` for house style only; do not invoke publication or its seam checkpoint. A diagnostic or audit determination — *assess, compare, or diagnose* something that already exists (has X diverged? why? what to do?) rather than pick or generate a plan — is a further sub-case: there is no candidate set, so candidate choice is N/A, the deliverable is a findings verdict plus recommendation that the closing recap carries, and the minimum design tree reduces to scope boundary and deliverable depth — a floor, not a ceiling: a finding that exposes a divergence between the user's stated intent and the repo's recorded authority (a principle, package rule, or spec encoding a mission the user no longer holds) re-opens the tree, and is asked as a ratification branch before the recap, since artifact-vs-artifact checks cannot settle intent. Expect intake to dominate the session — broad reads and parallel subagents are legitimate here and are how the "explore instead of ask" rule below scales — which resolves, in the audit direction, the apparent tension between this paragraph's "minimal intake" for cited-artifact stress-tests and the "broader intake" a determination invites.

Recap contracts: which recap a run owes, and what must be in it, is defined in `references/recap-contracts.md`. Route by class using its routing table — determination runs owe the PRD-ready/issue-ready checklist, diagnostic runs owe the diagnostic checklist plus the final preflight, and operational runs always owe the operational recap at minimum. "Keep the ledger minimal" bounds the branch count, never the recap; no class is recap-less. Applicability is decided by the run's *subject*, not by the kind of evidence used to investigate it: a run whose subject is a durable artifact (a report, PRD, issue, spec, or architecture review) but whose action mutates live state owes the operational recap **plus** the durable-artifact provenance fields, not a choice between the two. When the class shifts mid-session, the recap owed is the union of the contracts for every class the run actually occupied. Use `Finding:` or `Explored fact:` lines for factual conclusions and evidence-backed observations; reserve `Decision:` lines for actual user-owned stewardship or design choices.

Ask the questions one at a time, waiting for feedback on each before continuing — asking multiple at once is bewildering. When you pose a question, follow this format checklist:

- **Recommended option first, and append "(Recommended)" to that option's `label` field itself** — not only in the framing prose — so the marker renders on the option chip.
- A question may carry one tightly-coupled sub-decision when both share a single decision surface; never bundle independent branches.
- Ask via the available and permitted question tool (for example, AskUserQuestion where present); fall back to a prose question when the tool is absent, mode-restricted, disallowed by the active harness, or otherwise unavailable. If the active mode or harness says the tool is unavailable, Plan-only, or otherwise not callable, do not probe-call it — use the prose fallback shape directly.
- When options represent concrete artifacts — code shapes, file layouts, deliverable structures, config — use the tool's `preview` field to render each for side-by-side comparison; reserve previews for genuinely structural or visual choices, not simple preference questions.
- Prose fallback shape: `Question N: <decision surface>. My recommendation (Recommended): <answer and why>. Do you agree?`

If higher-priority response requirements add a footer, citation block, or other mandatory trailing content, keep the decision question immediately before that required trailing content and do not add a second prompt after it. When only prose questions are available, timeout/autoresolution behavior is unavailable: ask one question and wait for a reply unless the session is explicitly autonomous, in which case resolve remaining branches with recommended answers marked PROVISIONAL and recap them. During ordinary branch questions, a user reply like "proceed with your recommendation", "as you recommend", or equivalent clear acceptance of the recommended answer ratifies the current recommended answer and advances to the next unresolved branch; it does not skip the final recap checkpoint. A reply that accepts the recommendation while adding a correction, constraint, or free-text refinement ratifies the branch and additionally spawns its own ledger line for the amendment; restate the amendment in the next question's framing so the user can veto the interpretation. If the user replies with a clarification request, challenge, or "why" question instead of an answer, answer that reply directly, revise the recommendation if the answer changes the branch, do not ledger or advance the branch yet, and restate or await confirmation of the same decision surface. If the user explicitly delegates all remaining branches (for example, "use your recommendations for the rest"), resolve the remaining branches with recommended answers marked PROVISIONAL, present the recap with those decisions open to veto, and still require the final recap confirmation before any downstream deliverable. Keep the framing before a question to a few sentences — supporting detail can wait for the recap. If the first question times out right after a long context dump — or a later question times out moments after the user actively answered the previous one — the user may have been reading, or briefly stepped away, rather than gone; weigh re-asking that single question once before entering the fallback. Recent active engagement is the signal here, independent of question position. When the session context already states the user is away or the run is autonomous, skip the re-ask and enter the fallback directly — the batched round is the first ask worth a blocking wait (exploration-eligible branches still resolve by exploration first). An explicit grilling invocation is itself consent to blocking questions: generic autonomy framing in the session context (for example, a harness reminder that questions block work) does not suppress live questioning — ask live first and let the timeout fallback handle actual absence. Treat only an explicit statement that the user is away, or a scheduled/unattended run, as grounds to go provisional pre-emptively.

If a question times out with no response (the user is away), fall back once: batch the remaining decision branches into a single multi-question round up to the available question tool's current limit; never exceed the active tool schema. Prioritize the branches with the largest downstream impact — this is the sole exception to the no-bundling rule, since each retry costs a blocking wait. The question that triggered the fallback is itself a remaining branch: re-include it when it is among the largest-downstream-impact branches (as the upstream root it usually is), rather than treating its timeout as resolution. Branches that don't fit the round are resolved as if it had gone unanswered (recommended answer, PROVISIONAL), or by exploration where eligible. Phrase downstream questions in the batch conditionally on the recommended upstream answer; a vetoed upstream answer voids its dependent ledger lines. If the round also goes unanswered, apply your recommended answer, mark the ledger line PROVISIONAL, and continue. If instead the batch is answered, treat that as re-engagement: resume normal one-at-a-time questioning for any branches still open, and upgrade any branch that was defaulted PROVISIONAL only because it did not fit the round into a live question rather than leaving it provisional. The closing recap must list provisional decisions separately as open to veto; proceed to the deliverable only if it was explicitly requested up front — otherwise stop at the recap. When the request asks for a determination or recommendation, the recap containing it satisfies the deliverable; if no downstream artifact or action was explicitly requested, close on that recap without an extra final confirmation. Producing downstream artifacts (documents, issues, code) still requires an explicit request.

If a question can be answered by exploring the codebase or repo authorities, explore first instead. Repo authorities include package docs, principles, specs, ADRs, glossary/context files, tracker issues, and implementation surfaces. When a branch turns on whether the docs intend a behavior, run a targeted authority check before asking. Record the explored answer as a RATIFIED ledger line when the facts are conclusive and the answer resolves a design branch. When an explored fact materially shapes later decisions but is not itself a design choice, either include a short `Explored facts:` block before the recap, tied to the decisions it shaped, or use `Finding:` / `Explored fact:` lines in diagnostic and audit determinations. Do not re-ask it as a branch question unless the explored facts still leave a design or stewardship choice unresolved; in that case, ask only that unresolved choice and cite the explored fact in the rationale. If the explored answer may be sensitive to user intent rather than repo truth, include it in the recap as open to correction instead of spending a separate branch question on it.

As each question is resolved, append the decision to a running ledger (one line: status, question, answer, rationale) and derive the closing recap from it. For short in-context sessions the running ledger may be maintained internally and surfaced in full at the recap; emit an interim `Decision:` line per answer as you go only when the session is long enough to warrant the scratchpad file noted below. Either way, use this exact compact line format for every ledger line, and preserve it when the closing recap restates those decisions — no compressed variants:

```
Decision: [RATIFIED|PROVISIONAL] <question> -> <answer>; rationale: <why this answer wins>
```

Examples:

```
Decision: RATIFIED severity boundary -> neutral thresholds stay shared; rationale: other flows need the same skip threshold.
Decision: PROVISIONAL ADR need -> create one short ADR; rationale: the user is away and the trade-off is hard to reverse.
```

Mark RATIFIED when the user answered or confirmed the decision, or when exploration conclusively answers a factual branch. Mark PROVISIONAL when the answer came from timeout fallback, autonomous execution, or your recommended answer without explicit user ratification. Keep the ledger in-conversation for short sessions; create a scratchpad file only when the branch count or session length makes in-context tracking unreliable. If a supporting skill concludes that no update is owed, do not turn that into a decision ledger line such as `Decision: RATIFIED domain model unchanged`; report it separately as `Supporting skill result: Domain model unchanged...` unless the user actually ratified a domain or stewardship choice. If context compacts or the session otherwise resumes mid-interview, reconstruct the ledger from available conversation state, revalidate cited artifacts and current files as needed, restate the known decisions, and continue from the shallowest unresolved branch.

Before producing any downstream deliverable (documents, issues, code, implementation, or other artifacts), the recap is a hard checkpoint whenever at least one RATIFIED stewardship or design decision shaped the plan. This applies even when the deliverable was requested up front. Explored facts recorded as RATIFIED do not by themselves trigger a blocking confirmation in delegated operational requests; if the user explicitly requested the deliverable or action up front and no unresolved user-owned stewardship choice remains, record those facts in the final recap and proceed with the requested work. Mandatory inline updates required by a supporting skill are not downstream deliverables for this checkpoint: when a supporting skill explicitly requires immediate writes as decisions crystallize, such as `/domain-modeling` updating `CONTEXT.md` for a resolved term, make only the required minimal update and list it in the recap before any optional deliverable — but only once every branch governing whether standing files change this session (deliverable depth, mutation scope) is resolved. While such a branch is open, an inline write would preempt a user-owned choice: hold the resolved terms aside and list them in the recap as pending writes, the same hold-aside shape `/domain-modeling` uses while its glossary file's own shape is undecided. When a supporting skill concludes no update is owed, report that conclusion under a supporting-skill result line, not as a `Decision: RATIFIED ...` ledger entry, unless the user actually ratified a domain or stewardship choice. Show the ratified decisions, list any PROVISIONAL decisions separately, and ask the final confirmation as the last question (for example, "Proceed with <deliverable> on this basis?"). Do not start the deliverable until the user confirms; an explicit follow-up like "proceed" or "implement this" satisfies the confirmation. In a diagnose-and-fix or audit-plus-edit hybrid where the user requested the edit up front, when the final resolved branch is itself an AskUserQuestion or prose fallback question whose selected option or accepted recommendation already enumerates the deliverable's shape (for example an edit-strategy option that lists the exact sections to change), that selection or confirmation satisfies this final checkpoint — proceed without a separate "proceed?" prompt. This carve-out covers only a terminal option-selection or prose confirmation that already commits the deliverable's shape; it does not loosen the mid-interview rule that accepting a recommendation advances the branch but never skips the final checkpoint.

Autonomous deliverables exception: when the run is explicitly autonomous or the user delegated execution up front, and every unresolved branch was resolved as PROVISIONAL, present the provisional recap as a record, list the decisions that remain open to veto, and then proceed only with the deliverable or action the user explicitly requested. Do not create extra downstream artifacts from provisional decisions.

When every branch is resolved and a downstream deliverable or action was explicitly requested, apply the delegated-operational exception above when it fits; otherwise close with a recap of the ratified decisions (listing any PROVISIONAL ones separately) and the resulting deliverable, confirm it as the final question, then proceed. During execution, edits mechanically required for the confirmed deliverable's internal consistency — version markers, index rows, derived-document re-syncs the edited artifact's own rules demand — are in scope without a fresh confirmation, provided each traces to a ratified decision or the artifact's own consistency discipline; enumerate them in the post-deliverable summary so the expansion is visible. For deliverables that write repo files in a long session, re-check worktree currency at **two** mandatory points: **(1)** before the first write, and **(2)** again immediately before the post-deliverable summary — re-run `git status --short` and `git branch --show-current` each time, and do not assert worktree state in the summary from the pre-write snapshot alone. File-write checklist: before first write, run `git status --short` and `git branch --show-current`; before the final summary, run `git status --short` and `git branch --show-current` again. This checklist governs local writes; a versioned repo file committed directly on the remote instead follows the server-side-commit case in the operational guidance above — the worktree checklist is N/A, and the end state is proven by a remote read-back. When the confirmed deliverable includes code, run the repository's canonical verification gates before the post-deliverable summary — read the project's verification guidance for the exact commands — and report each as passed, failed, or unavailable with a reason; never claim an implementation complete on unrun gates. When the deliverable mutated a tracker or other external surface, re-read the affected items before the summary and show the read-back that proves the intended end state. If the baseline moved mid-session — branch switched, HEAD advanced, or session edits were absorbed into external commits — verify the committed content still matches the session's edits and restate the corrected worktree state in the summary rather than repeating the recap-time snapshot. If no downstream deliverable or action was explicitly requested, close on the recap itself; when the ratified design clearly implies a useful next artifact, include one short non-actioning note in the recap, such as "This is PRD-ready", "This is issue-ready", or "This may warrant an ADR". For runs that close as PRD-ready or issue-ready, carry the provenance clauses from `references/recap-contracts.md` — source artifact, selected section, candidate set, winning candidate, key rejected alternatives, tracker IDs, and inspected authority paths — so a later `/to-prd` or `/to-issues` pass does not have to reconstruct them from conversation context. Do not create the artifact or ask a second artifact-specific question unless the user requests it. Skip confirmation when no question in the session was answered or every answered branch is PROVISIONAL; in that case, close on the recap itself, open to veto. If the user explicitly requested a downstream deliverable or action at the start, the autonomous deliverables exception above controls whether to proceed.

If the session resumes after a provisional close, treat each veto as re-opening that ledger line plus every line that depended on it, then resume one-at-a-time questioning from the shallowest re-opened branch; untouched lines stand as ratified.
