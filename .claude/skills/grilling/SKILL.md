---
name: grilling
description: Interview the user relentlessly about a plan or design. Use when the user wants to stress-test a plan before building, or uses any 'grill' trigger phrases.
---

Interview me relentlessly about every aspect of this plan until we reach a shared understanding. Walk down each branch of the design tree, resolving dependencies between decisions one-by-one. For each question, provide your recommended answer.

Before asking, classify the request. If it is a design or plan stress-test, run the full interview. If it is an operational, triage, dependency-maintenance, or other delegated-execution request that uses grilling to sharpen judgment, explore the live facts first, resolve only the decision branches that materially affect the action, keep the ledger minimal, and proceed with the explicitly requested work once those branches are resolved. Do not turn an operational request into a broad design interview unless the user asks for that.

For design stress-tests that cite a report, module, architecture review, artifact, or repo authority, do a minimal intake before the first question: open the cited artifact, read the relevant glossary/principles/ADRs, inspect the current implementation surface, and turn only unresolved stewardship or design choices into questions. Keep this intake just deep enough to avoid asking what repo truth already answers. When the request is instead to *determine, choose, or generate* a plan rather than stress-test one that already exists (for example, "decide the next PRD"), there is no plan yet to grill: intake is legitimately broader — survey the surface to assemble the candidate set — and the root question becomes the choice among those candidates, after which you grill the winning candidate's design tree one branch at a time. If the repo declares a tracker, backlog, PRD, issue, or run-state surface, include the current open items and latest relevant closed parent/PRD in that survey so the candidate set does not duplicate or contradict live work. For determination/recommendation runs, the minimum design tree is usually candidate choice, scope boundary, and deliverable depth; once those are resolved, stop at a PRD-ready or issue-ready recap unless the user explicitly requested publication, issue creation, document writing, or implementation. This is the same determination/recommendation case the deliverable rule below already handles; the recap still carries the choice as its determination.

Ask the questions one at a time, waiting for feedback on each question before continuing. Asking multiple questions at once is bewildering. A question may carry one tightly-coupled sub-decision when both share a single decision surface; never bundle independent branches. Ask each question via the available and permitted question tool (for example, AskUserQuestion where present) — recommended option first, with "(Recommended)" appended to that option's label field itself — not only stated in the framing prose, so the marker shows on the option chip; fall back to a prose question when the tool is absent, mode-restricted, disallowed by the active harness, or otherwise unavailable. When a question's options represent concrete artifacts — code shapes, file layouts, deliverable structures, config — use the question tool's `preview` field to render each option for side-by-side comparison; reserve previews for genuinely structural or visual choices, not simple preference questions. When falling back to prose, use this shape: `Question N: <decision surface>. My recommendation (Recommended): <answer and why>. Do you agree?` When only prose questions are available, timeout/autoresolution behavior is unavailable: ask one question and wait for a reply unless the session is explicitly autonomous, in which case resolve remaining branches with recommended answers marked PROVISIONAL and recap them. During ordinary branch questions, a user reply like "proceed with your recommendation", "as you recommend", or equivalent clear acceptance of the recommended answer ratifies the current recommended answer and advances to the next unresolved branch; it does not skip the final recap checkpoint. If the user explicitly delegates all remaining branches (for example, "use your recommendations for the rest"), resolve the remaining branches with recommended answers marked PROVISIONAL, present the recap with those decisions open to veto, and still require the final recap confirmation before any downstream deliverable. Keep the framing before a question to a few sentences — supporting detail can wait for the recap. If the first question times out right after a long context dump — or a later question times out moments after the user actively answered the previous one — the user may have been reading, or briefly stepped away, rather than gone; weigh re-asking that single question once before entering the fallback. Recent active engagement is the signal here, independent of question position. When the session context already states the user is away or the run is autonomous, skip the re-ask and enter the fallback directly — the batched round is the first ask worth a blocking wait (exploration-eligible branches still resolve by exploration first).

If a question times out with no response (the user is away), fall back once: batch the remaining decision branches into a single multi-question round up to the available question tool's current limit; never exceed the active tool schema. Prioritize the branches with the largest downstream impact — this is the sole exception to the no-bundling rule, since each retry costs a blocking wait. The question that triggered the fallback is itself a remaining branch: re-include it when it is among the largest-downstream-impact branches (as the upstream root it usually is), rather than treating its timeout as resolution. Branches that don't fit the round are resolved as if it had gone unanswered (recommended answer, PROVISIONAL), or by exploration where eligible. Phrase downstream questions in the batch conditionally on the recommended upstream answer; a vetoed upstream answer voids its dependent ledger lines. If the round also goes unanswered, apply your recommended answer, mark the ledger line PROVISIONAL, and continue. If instead the batch is answered, treat that as re-engagement: resume normal one-at-a-time questioning for any branches still open, and upgrade any branch that was defaulted PROVISIONAL only because it did not fit the round into a live question rather than leaving it provisional. The closing recap must list provisional decisions separately as open to veto; proceed to the deliverable only if it was explicitly requested up front — otherwise stop at the recap. When the request asks for a determination or recommendation, the recap containing it satisfies the deliverable; if no downstream artifact or action was explicitly requested, close on that recap without an extra final confirmation. Producing downstream artifacts (documents, issues, code) still requires an explicit request.

If a question can be answered by exploring the codebase, explore the codebase instead. Record the explored answer as a RATIFIED ledger line when the facts are conclusive. When an explored fact materially shapes later decisions but is not itself a design choice, either record it as a RATIFIED factual ledger line or include a short `Explored facts:` block before the recap, tied to the decisions it shaped. Do not re-ask it as a branch question unless the explored facts still leave a design or stewardship choice unresolved; in that case, ask only that unresolved choice and cite the explored fact in the rationale. If the explored answer may be sensitive to user intent rather than repo truth, include it in the recap as open to correction instead of spending a separate branch question on it.

As each question is resolved, append the decision to a running ledger (one line: status, question, answer, rationale) and derive the closing recap from it. Use this exact compact line format: `Decision: [RATIFIED|PROVISIONAL] <question> -> <answer>; rationale: <why this answer wins>`. Examples: `Decision: RATIFIED severity boundary -> neutral thresholds stay shared; rationale: other flows need the same skip threshold.` `Decision: PROVISIONAL ADR need -> create one short ADR; rationale: the user is away and the trade-off is hard to reverse.` Mark RATIFIED when the user answered or confirmed the decision, or when exploration conclusively answers a factual branch. Mark PROVISIONAL when the answer came from timeout fallback, autonomous execution, or your recommended answer without explicit user ratification. Keep the ledger in-conversation for short sessions; create a scratchpad file only when the branch count or session length makes in-context tracking unreliable.

Before producing any downstream deliverable (documents, issues, code, implementation, or other artifacts), the recap is a hard checkpoint whenever at least one RATIFIED decision shaped the plan. This applies even when the deliverable was requested up front. Mandatory inline updates required by a supporting skill are not downstream deliverables for this checkpoint: when a supporting skill explicitly requires immediate writes as decisions crystallize, such as `/domain-modeling` updating `CONTEXT.md` for a resolved term, make only the required minimal update and list it in the recap before any optional deliverable. Show the ratified decisions, list any PROVISIONAL decisions separately, and ask the final confirmation as the last question (for example, "Proceed with <deliverable> on this basis?"). Do not start the deliverable until the user confirms; an explicit follow-up like "proceed" or "implement this" satisfies the confirmation.

Autonomous deliverables exception: when the run is explicitly autonomous or the user delegated execution up front, and every unresolved branch was resolved as PROVISIONAL, present the provisional recap as a record, list the decisions that remain open to veto, and then proceed only with the deliverable or action the user explicitly requested. Do not create extra downstream artifacts from provisional decisions.

When every branch is resolved and a downstream deliverable or action was explicitly requested, close with a recap of the ratified decisions (listing any PROVISIONAL ones separately) and the resulting deliverable, confirm it as the final question, then proceed. If no downstream deliverable or action was explicitly requested, close on the recap itself; when the ratified design clearly implies a useful next artifact, include one short non-actioning note in the recap, such as "This is PRD-ready", "This is issue-ready", or "This may warrant an ADR". Do not create the artifact or ask a second artifact-specific question unless the user requests it. Skip confirmation when no question in the session was answered or every answered branch is PROVISIONAL; in that case, close on the recap itself, open to veto. If the user explicitly requested a downstream deliverable or action at the start, the autonomous deliverables exception above controls whether to proceed.

If the session resumes after a provisional close, treat each veto as re-opening that ledger line plus every line that depended on it, then resume one-at-a-time questioning from the shallowest re-opened branch; untouched lines stand as ratified.
