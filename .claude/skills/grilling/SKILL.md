---
name: grilling
description: Interview the user relentlessly about a plan or design. Use when the user wants to stress-test a plan before building, or uses any 'grill' trigger phrases.
---

Interview me relentlessly about every aspect of this plan until we reach a shared understanding. Walk down each branch of the design tree, resolving dependencies between decisions one-by-one. For each question, provide your recommended answer.

Ask the questions one at a time, waiting for feedback on each question before continuing. Asking multiple questions at once is bewildering. A question may carry one tightly-coupled sub-decision when both share a single decision surface; never bundle independent branches. Ask each question via the available question tool (for example, AskUserQuestion where present) — recommended option first, labeled "(Recommended)"; fall back to a prose question otherwise. When only prose questions are available, timeout/autoresolution behavior is unavailable: ask one question and wait for a reply unless the session is explicitly autonomous, in which case resolve remaining branches with recommended answers marked PROVISIONAL and recap them. During ordinary branch questions, a user reply like "proceed with your recommendation", "as you recommend", or equivalent clear acceptance of the recommended answer ratifies the current recommended answer and advances to the next unresolved branch; it does not skip the final recap checkpoint. Keep the framing before a question to a few sentences — supporting detail can wait for the recap. If the first question times out right after a long context dump, the user may have been reading rather than away; weigh re-asking once before entering the fallback. When the session context already states the user is away or the run is autonomous, skip the re-ask and enter the fallback directly — the batched round is the first ask worth a blocking wait (exploration-eligible branches still resolve by exploration first).

If a question times out with no response (the user is away), fall back once: batch the remaining decision branches into a single multi-question round up to the available question tool's current limit; never exceed the active tool schema. Prioritize the branches with the largest downstream impact — this is the sole exception to the no-bundling rule, since each retry costs a blocking wait. The question that triggered the fallback is itself a remaining branch: re-include it when it is among the largest-downstream-impact branches (as the upstream root it usually is), rather than treating its timeout as resolution. Branches that don't fit the round are resolved as if it had gone unanswered (recommended answer, PROVISIONAL), or by exploration where eligible. Phrase downstream questions in the batch conditionally on the recommended upstream answer; a vetoed upstream answer voids its dependent ledger lines. If the round also goes unanswered, apply your recommended answer, mark the ledger line PROVISIONAL, and continue. The closing recap must list provisional decisions separately as open to veto; proceed to the deliverable only if it was explicitly requested up front — otherwise stop at the recap. When the request asks for a determination or recommendation, the recap containing it satisfies the deliverable; producing downstream artifacts (documents, issues, code) still requires an explicit request.

If a question can be answered by exploring the codebase, explore the codebase instead.

As each question is resolved, append the ratified decision to a running ledger (one line: question, answer, rationale) and derive the closing recap from it. Use a compact line format: `Decision: <question> -> <answer>; rationale: <why this answer wins>`. Keep the ledger in-conversation for short sessions; create a scratchpad file only when the branch count or session length makes in-context tracking unreliable.

Before producing any downstream deliverable (documents, issues, code, implementation, or other artifacts), the recap is a hard checkpoint whenever at least one decision was answered. This applies even when the deliverable was requested up front. Show the ratified decisions, list any PROVISIONAL decisions separately, and ask the final confirmation as the last question (for example, "Proceed with <deliverable> on this basis?"). Do not start the deliverable until the user confirms; an explicit follow-up like "proceed" or "implement this" satisfies the confirmation.

When every branch is resolved, close with a recap of the ratified decisions (listing any PROVISIONAL ones separately) and the resulting deliverable, confirm it as the final question, then proceed. Skip that confirmation when no question in the session was answered — an all-provisional session closes on the recap itself, open to veto, rather than spending another blocking wait.

If the session resumes after a provisional close, treat each veto as re-opening that ledger line plus every line that depended on it, then resume one-at-a-time questioning from the shallowest re-opened branch; untouched lines stand as ratified.
